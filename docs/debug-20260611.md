## Sekaiemu GBA BizHawk Substitute Bridge

- Status: fix implemented, awaiting fresh live Fusion retest.
- Reference imported:
  - `third_party/bhc-substitutes/mgba/connector_bizhawkclient_mgba.lua`
  - `third_party/bhc-substitutes/pj64/connector_bizhawkclient_pj64.js`
- Finding:
  - Zunawe's mGBA substitute exposes GBA `System Bus` by delegating directly to
    mGBA's emulator bus object. This confirms that generic BizHawk-compatible
    GBA clients expect bus addresses such as `0x0E01FFFE` to behave like real
    GBA bus addresses, not only like flat libretro memory offsets.
  - The live Sekaiemu/mGBA run exposed 32 KiB save RAM, while the previous smoke
    test only covered a 128 KiB save RAM fixture. Fusion's receive counter uses
    `System Bus` addresses `0x0E01FFFE` and `0x0E01FFFF`, which need save/SRAM
    mirroring when the core exposes a smaller save RAM region.
- Fix:
  - `runtime_memory_server.cpp` now mirrors high GBA save/SRAM bus offsets into
    the exposed libretro save RAM region for both `System Bus` and `SRAM`.
  - `runtime_memory_server_bizhawk_protocol_smoke.cpp` now verifies the 32 KiB
    mirrored case for reads and writes.
- Verification:
  - `cmake --build /tmp/sekaiemu-gba-bridge-build --target runtime_memory_server_bizhawk_protocol_smoke sekaiemu_libretro_spike -j$(nproc)`
  - `/tmp/sekaiemu-gba-bridge-build/runtime_memory_server_bizhawk_protocol_smoke`
  - Result: `sekaiemu_runtime_memory_server_bizhawk_protocol_smoke_ok`

    `IWRAM:offset -> System Bus 0x03000000 + offset`, and
    `SRAM:offset -> System Bus 0x0E000000 + offset`.
  - Extended the BizHawk protocol smoke test to cover split GBA IWRAM via the
    libretro memory map.
  - Rebuilt and staged the Linux Sekaiemu runtime binary in both local runtime
    paths used by Client Core.

  - Direct socket reads during Fusion showed Morph Ball state in `IWRAM`.
  - The AP client sent `LocationChecks (1): Main Deck -- Quarantine Bay`.
  - The server responded with `ReceivedItems` and `RoomUpdate`.
- Follow-up:
