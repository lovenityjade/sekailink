# Debug 2026-06-26

## GBA check routing

Primary directive: do not debug by attacking games one by one when the failing behavior is actually a core/bridge/runtime contract issue. Games are still used as witnesses, but the target is stable memory exposure, SKLMI/client read/write, and AP server send/receive.

Current GBA witness status:

- The Minish Cap is confirmed bilateral. This proves the GBA core, memory bridge, AP server connection, and direct `LocationChecks` path can work.
- Pokemon Emerald connects, receives items, sends tracker/map `Bounce`/`Set` packets, but did not send checks in the observed run.
- Wario Land 4 is confirmed bilateral. Server hints, `LocationScouts`, `Set`/`Bounce`, item receipt, and check flow were confirmed after accounting for its delayed check behavior.

Working hypothesis:

- This is not a global GBA failure.
- Some GBA clients use direct `send_msgs({"cmd": "LocationChecks"})`; The Minish Cap uses that path and works.
- Some GBA clients use internal state and/or `ctx.check_locations`; Pokemon Emerald uses `ctx.check_locations`, while Wario Land 4 directly sends `LocationChecks` only when its local collected-location set changes.
- Wario Land 4 defaults `send_locations_to_server` to `on_escape`, so pickups inside a level may not be sent until escape/level exit unless the generated option is set to `immediately`.

Instrumentation added:

- `runtime/apclient_common.py`
  - Emits `connected_slot_data` so generated settings/options are visible in logs for every AP client run.
  - Emits whether `ctx.check_locations` is hooked.
  - Emits `check_locations_call` with requested IDs and IDs that would be accepted by `missing_locations`.
  - Adds `handler_local_checked_count` to `bizhawk_watcher_tick`.
- `runtime/ap/worlds/pokemon_emerald/client.py`
  - Emits `pokemon_emerald_location_debug` with set flag count, matched AP location count, event count, key item flag count, pokedex byte count, saveblock pointers, and server location count.
- `runtime/ap/worlds/wl4/client.py`
  - Emits `wl4_location_debug` with inventory-derived location count, live pickup location count, pending scout count, outgoing location count, and outgoing packet types.

Next validation:

Debug rule added after testing:

- Always verify the generated game settings before treating a missing check as a runtime bug. Some supported games delay checks, disable certain check categories by default, or only send locations after a specific state transition.
- Pokemon Emerald remains a suspected real bug for the current run because the tested action was confirmed by Jade as an expected check.

## MMBN3 runtime client

- Mega Man Battle Network 3 is not a generic GBA BizHawk-client game in the bundled Archipelago source.
- Its official setup uses `MMBN3Client.py` plus `data/lua/connector_mmbn3.lua`; the Lua connector listens on `localhost:28922` and exchanges JSON payloads with the client.
- The previous SekaiLink metadata claimed `ap_client = bizhawk` while the registry marked the game as `generator_only`, so autolaunch patched and launched Sekaiemu but refused to start the AP client with `archipelago_client_runtime_unavailable`.
- Added the official `runtime/ap/MMBN3Client.py` and a SekaiLink `runtime/mmbn3_bridge.py` that emulates the read side of `connector_mmbn3.lua` against Sekaiemu's memory socket.
- Updated the MMBN3 module and client registry to use `client_mode = mmbn3` / `wrapper = mmbn3`.
- Validation passed: Python compile, JSON validation, and Electron `node --check`.
- Current availability: disabled/not available until a dedicated MMBN3 tester can validate the bridge flow.
- Next live test, when re-enabled, should confirm that `mmbn3_bridge_ready`, `mmbn3_bridge_client_connected`, and MMBN3 `Connected` appear before evaluating item/check R/W behavior.
